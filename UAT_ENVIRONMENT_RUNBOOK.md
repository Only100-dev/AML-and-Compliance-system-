## 3. How to Reset to Fresh UAT State

If the UAT environment becomes corrupted (e.g., a tester accidentally modifies seed data, or a bug fix requires a clean re-seed), follow this procedure to reset to a known-good fresh state.

**⚠️ Warning**: This procedure deletes ALL data in the dev database, including load-test data. The load-test data can be regenerated with `node scripts/generate-mock-data.mjs`, but any in-flight UAT test progress will be lost. Coordinate with the Compliance Manager before resetting.

```bash
cd /home/z/my-project

# Step 1: Restore the frozen schema (only if schema.prisma was modified)
cp prisma/schema.frozen-v7.3.0.prisma prisma/schema.prisma

# Step 2: Wipe the database (SQLite files)
rm -f db/custom.db db/custom.db-shm db/custom.db-wal

# Step 3: Push the schema to create a fresh empty database
bun run db:push

# Step 4: (Optional) Regenerate load-test data for realistic scale
# This populates ~18,500 records (complaints, CAPs, audit logs, etc.)
node scripts/generate-mock-data.mjs

# Step 5: Seed the 62 UAT records (idempotent — safe to re-run)
bunx tsx prisma/seed-uat.ts

# Step 6: (Optional) Backfill UniversalTasks from existing alerts/complaints/CAPs/SARs
node scripts/backfill-universal-tasks.mjs

# Step 7: (Optional) Backfill audit hashes if the hash formula changed
node scripts/backfill-audit-hashes.mjs

# Step 8: Restart the dev server (kill the old process first if running)
bun run dev
```

**Verify the reset**:
```bash
# Should return valid: true with ~10,727+ entries (or ~62 if you skipped load-test data)
curl "http://localhost:3000/api/audit/integrity?fresh=1"

# Should return 3 tasks for the Compliance Manager
curl -H "x-user-role: compliance_manager" -H "x-user-id: cmqm1payy0002orjaeg2irv2e" \
     http://localhost:3000/api/tasks/my-tasks
```

**Idempotency note**: `prisma/seed-uat.ts` is idempotent — if the sentinel user `uat-user-1@icos-test.local` already exists, the script logs a skip message and exits 0 without writing. To force a re-seed, you must wipe the database first (Step 2 above).

## 4. How to Verify Audit Integrity

The audit-trail integrity check is the single most important verification in UAT. Every UAT day should begin and end with this check. Any violation is a P0 bug.

### Quick Check (cached, ~100ms)
```bash
curl "http://localhost:3000/api/audit/integrity" \
  -H "x-user-role: compliance_manager" \
  -H "x-user-id: cmqm1payy0002orjaeg2irv2e"
```

### Fresh Check (bypass cache, ~100ms with cache, ~2s without)
```bash
curl "http://localhost:3000/api/audit/integrity?fresh=1" \
  -H "x-user-role: compliance_manager" \
  -H "x-user-id: cmqm1payy0002orjaeg2irv2e"
```

### Expected Response
```json
{
  "success": true,
  "data": {
    "valid": true,
    "verifiedEntries": 10727,
    "totalEntries": 10727,
    "violations": [],
    "hashFormula": "SHA-256(JSON.stringify({ userId, action, resource, resourceId, details, createdAt }))",
    "lastVerifiedAt": "2026-...",
    "cacheHit": false
  }
}
```

### What Each Field Means
- `valid: true` — All entries' stored hashes match the recomputed hashes. The audit trail is tamper-free.
- `verifiedEntries` — Number of entries whose hash was recomputed and matched. Should equal `totalEntries`.
- `totalEntries` — Total number of AuditLog rows in the database.
- `violations` — Array of entries that failed verification. **Must be empty.** If non-empty, each violation includes `entryId`, `expectedHash`, `actualHash`, and `reason`.
- `hashFormula` — The exact SHA-256 formula used. Inspect `src/lib/audit.ts::createAuditLog()` and `src/app/api/audit/integrity/route.ts` to confirm both sides use the same formula.

### If `valid: false`
1. **Halt UAT** — audit-trail integrity is a hard regulatory requirement.
2. **Escalate to Development Team lead** (P0, 4-hour fix SLA).
3. **Do NOT attempt to fix hashes manually** — the root cause is usually a `createAuditLog` call that hashes different fields than the integrity endpoint expects, or a record written outside `createAuditLog`.
4. **Run the backfill script** (only if the hash formula legitimately changed and a migration is in progress): `node scripts/backfill-audit-hashes.mjs`. This recomputes all hashes with the current formula. Use with caution — only the Dev Team lead should authorize this.

### Performance
- With cache (60s TTL, single-flight refresh): p95 < 100ms (verified in load tests).
- Without cache (fresh recomputation of 10,727 entries): p95 ~2 seconds. The `?fresh=1` query param bypasses the cache.
- The integrity endpoint is rate-limited at READ (100/min) — sufficient for UAT but not for high-frequency polling.

## 5. How to Run the SLA Evaluator

The SLA evaluator processes all eligible complaints, computes their SLA status (WITHIN_SLA / APPROACHING_BREACH / BREACHED), generates `ComplianceAlert` rows at the 80% (approaching) and 100% (breached) thresholds, and creates `UniversalTask` rows for the responsible Compliance Manager. CBUAE Notice 3551/2021 requires acknowledgment within 5 business days and resolution within 30 business days — the evaluator enforces these timelines.

### Manual Run (POST — evaluates and generates alerts/tasks)
```bash
curl -X POST http://localhost:3000/api/complaints/sla/evaluate \
  -H "x-user-role: compliance_manager" \
  -H "x-user-id: cmqm1payy0002orjaeg2irv2e"
```

### Expected Response (verified smoke-test data)
```json
{
  "success": true,
  "data": {
    "processedComplaints": 500,
    "alertsGenerated": 659,
    "statusesUpdated": 316,
    "breakdown": {
      "WITHIN_SLA": 120,
      "APPROACHING_BREACH": 180,
      "BREACHED": 200
    }
  }
}
```
*(Numbers vary based on the current dataset. The seed created 10 UAT complaints with 3 BREACHED, 3 APPROACHING_BREACH, 3 WITHIN_SLA, 1 N/A.)*

### Dashboard View (GET — returns current SLA state without re-evaluating)
```bash
curl http://localhost:3000/api/complaints/sla/evaluate \
  -H "x-user-role: compliance_manager" \
  -H "x-user-id: cmqm1payy0002orjaeg2irv2e"
```

### What the SLA Evaluator Does
1. Queries all complaints with `status NOT IN [CLOSED, ESCALATED_TO_OMBUDSMAN]`.
2. For each complaint, computes business-day duration since `createdAt` (using `src/lib/compliance/business-days.ts` — excludes UAE public holidays and weekends).
3. Compares against CBUAE thresholds:
   - Acknowledgment: 5 business days from `createdAt`
   - Resolution: 30 business days from `createdAt`
4. At 80% of threshold: generates `ComplianceAlert` with `alertType: COMPLAINT_SLA_APPROACHING`, severity `medium`.
5. At 100% of threshold: generates `ComplianceAlert` with `alertType: COMPLAINT_SLA_BREACHED`, severity `high` or `critical`.
6. For each alert, creates a `UniversalTask` assigned to the responsible Compliance Manager (linked via `assignedToId`), with `dueDate` set to the breach threshold.
7. All write operations are audit-logged via `createAuditLog`.

### Scheduled Execution (Production)
In production, the SLA evaluator is scheduled as a cron job (daily at 06:00 UAE time). In UAT, it is run manually as shown above. The cron route is at `src/app/api/cron/calculate-department-risk/route.ts` (department risk cron — the SLA evaluator cron is configured separately in `src/lib/cron/isolation.ts`).

## 6. How to Run PII Leak Detection

The PII leak detection script scans all API responses and database exports for raw PII patterns. It is the primary verification that PII masking is working correctly. **Zero raw PII matches is a hard UAT pass criterion.**

### Run the Script
```bash
cd /home/z/my-project
bunx tsx scripts/pii-leak-detection.mjs
```

### What the Script Checks
The script (`scripts/pii-leak-detection.mjs`) scans for the following PII patterns:
- **Emirates IDs** — format `xxx-xxxx-xxxx-x` (3-4-4-1 digits)
- **UAE phone numbers** — format `+971xxxxxxxxx` or `0xxxxxxxxxx`
- **IBANs** — format `AExxxxxxxxxxxxxxxxxxxxxx` (AE + 21 alphanumeric)
- **Email addresses** — except `@icos-test.local` test addresses (which are UAT-allowed)
- **Passport numbers** — alphanumeric 8–9 character sequences in context
- **Full names** — against a watchlist of known person names (if configured)

### Expected Output
```
PII Leak Detection Report
=========================
Scanned endpoints: 12
Scanned database exports: 8
Total patterns checked: 6
Raw PII matches: 0

✓ PASS — No raw PII detected in API responses or database exports.
```

### If Matches Are Found
- **Any match on real PII (Emirates ID, real phone, real IBAN, real email)**: **P0 bug**. Halt UAT. Escalate to Development Team lead immediately.
- **Match on `@icos-test.local` email**: Expected — these are UAT-allowed test addresses. No action.
- **Match on a masked pattern** (e.g., `***-****-****-1234`): Expected — masking is working. No action.

The script outputs each match with: the pattern type, the matched value (truncated), the source (endpoint or export), and the row ID. Use this to trace the leak to its source.

## 8. Known Issues / Notes

These are pre-existing known issues that UAT testers should be aware of. They are **not** UAT-blocking unless explicitly noted. Issues 1 and 2 were resolved in the v7.3.0-RC1-uat-ready blocker-fixes commit and are documented here for historical context only.

### Issue 1: `createUniversalTask()` helper field-name mismatch — **RESOLVED in v7.3.0-RC1-uat-ready**
**Location**: `src/lib/universal-task.ts`
**Status**: ✅ FIXED — The helper now uses the correct field names (`taskType`, `sourceId`, `sourceEntityType`, `sourceModule`). The helper is now wired into the CAP creation, Policy Attestation creation/acknowledgment, Dept-Head Circular acknowledgment, and Ingestion Commit fan-out routes. Verified by `scripts/verify-universal-task-helper.mjs` (5/5 tests pass: CAP assignment → task in inbox, attestation → task in inbox, acknowledgment → task removed, direct helper call → valid DB row with sha256Hash, idempotency).
**Historical context (pre-fix)**: The helper previously used `entityType`/`entityId` field names that did not match the Prisma model's `taskType`/`sourceId`/`sourceEntityType`. The helper had ZERO callers as a result — every flow that needed an inbox row wrote directly to `db.universalTask.create`, bypassing the helper. This meant CAPs, attestations, and circular acknowledgments did NOT surface in the unified My Tasks inbox. The fix rewires all three flows through the helper.

### Issue 2: Data-room endpoint — **RESOLVED in v7.3.0-RC1-uat-ready**
**Location**: `src/app/api/audit/generate-data-room/route.ts`
**Status**: ✅ FIXED — The endpoint now supports two response formats:
- `POST /api/audit/generate-data-room` → JSON response (default, backward-compatible)
- `POST /api/audit/generate-data-room?format=zip` → binary ZIP file download (new — regulator-distributable artifact)

The schema is now permissive: accepts `requestingUserName` OR `examinerName` (alias), optional `auditId` (nullable, filters `audit_logs` to one entry), optional `riskLevel` (default `all`). The ZIP contains `manifest.json` + 6 PII-masked CSVs + `integrity.txt` with SHA-256 verification instructions. Verified by both `scripts/pii-leak-detection.mjs` (JSON path) and `scripts/pii-leak-detection-zip.mjs` (ZIP path) — 0 raw PII matches in both.
**Historical context (pre-fix)**: The prior smoke test reported `success: false` because the test sent an incomplete request body (missing `requestJustification` which has a `min(20)` Zod constraint, or missing `requestingUserId`/`requestingUserName`). The endpoint itself was functional — the issue was incorrect test input, not a code defect.

### Issue 3: Turbopack cache corruption on broken imports
**Location**: `.next/` (Turbopack build cache)
**Description**: The dev server's Turbopack cache can become corrupted if a route has a broken import (e.g., the P0 SLA evaluator import bug that was just fixed). Symptoms: multiple endpoints return HTTP 500 unexpectedly, even endpoints unrelated to the broken import.
**Resolution**: Clear the `.next` directory and restart the dev server:
```bash
cd /home/z/my-project
rm -rf .next
bun run dev
```
**UAT Impact**: If multiple endpoints start returning 500 unexpectedly during UAT, try clearing `.next` first before reporting a bug — the issue may be cache corruption rather than a real defect. If the 500s persist after clearing `.next`, then it's a real bug.
**Prevention**: The Development Team should always clear `.next` after pulling code changes or switching branches.

### Issue 4: `board` role cannot access My Tasks inbox (by design)
**Description**: The `board` role does not have the `canViewUnifiedTasks` permission and receives a 403 when calling `GET /api/tasks/my-tasks`. This is **by design** — board members use the Board Portal (`src/components/ic-os/board/BoardPortal.tsx`) for read-only dashboard access, not the operational my-tasks inbox.
**UAT Impact**: When testing Scenario 5 with the `board` account (account #6), expect a 403 on `/api/tasks/my-tasks`. This is the correct behavior — verify the 403 is returned and move on. Do NOT log as a bug.

### Issue 5: Load-test data coexists with UAT data
**Description**: The UAT environment contains ~18,500 load-test records (from `scripts/generate-mock-data.mjs`) alongside the 62 UAT records. This is intentional — it provides realistic scale for performance verification. However, it means list endpoints (e.g., `GET /api/complaints`) return a mix of UAT-prefixed (`UAT-CMP-2026-NNNN`) and load-test-prefixed (`CMP-2026-*`) records.
**UAT Impact**: When verifying Scenario 1, filter the complaints list by `complaintNumber` starting with `UAT-` to isolate the UAT records. The 10 UAT complaints have predictable SLA states (3 BREACHED, 3 APPROACHING_BREACH, 3 WITHIN_SLA, 1 N/A) — use these for deterministic testing.

### Issue 6: Audit integrity discrepancy (2 null-hash entries) — **RESOLVED in v7.3.0-RC1-uat-final**
**Location**: `src/app/api/attestations/route.ts` (POST + PUT handlers)
**Status**: ✅ FIXED — The attestations route now creates AuditLog entries via the canonical `createAuditLog()` helper (`src/lib/audit.ts`), which computes the SHA-256 integrity hash + sanitizes PII. Previously, both handlers called `db.auditLog.create()` directly, bypassing hash computation — producing entries with a NULL `sha256Hash` that the `/api/audit/integrity` verifier counts as `missingHashCount` (not `violations`, but they depress `verifiedEntries` below `totalEntries`).
**Discrepancy details (pre-fix)**: Pre-UAT smoke testing on 2026-06-20 ~08:50 UTC reported `11395/11397 verified` — 2 entries had NULL `sha256Hash`. Investigation (`scripts/investigate-audit-discrepancy.mjs`) identified both as `PolicyAttestation` actions (`ATTESTATION_CREATED` + `ATTESTATION_ACKNOWLEDGED`) created by the Blocker 2 verification script (`scripts/verify-universal-task-helper.mjs`) using policy number `UAT-POL-VERIFY-001`. They were test artifacts, not real compliance actions. The verification script's cleanup matched on the string `'UAT-VERIFY'`, which is NOT a substring of `'UAT-POL-VERIFY-001'` — so the 2 audit logs were orphaned while their associated attestation/task/calendar-event records were correctly deleted.
**Fix applied (3 parts)**:
1. **Root cause** — `src/app/api/attestations/route.ts` POST + PUT now call `createAuditLog()` instead of `db.auditLog.create()`. New attestations will always produce hashed audit entries. Verified end-to-end by `scripts/verify-attestation-audit-fix.mjs` (creates a test attestation → confirms both audit entries have non-null `sha256Hash` matching the verifier formula → re-checks integrity (`11399/11399`, 0 violations) → cleans up).
2. **Test-artifact deletion** — `scripts/cleanup-audit-discrepancy.mjs` deleted the 2 orphaned audit entries (surgical: only deletes NULL-hash entries whose `details` contain `UAT-POL-VERIFY-001`). Associated attestation/task/calendar records were already gone.
3. **Backfill safety net** — `scripts/backfill-audit-hashes.mjs` re-run (idempotent) confirms all 11397 entries match the verifier formula; 0 missing, 0 mismatched.
**Post-fix verification**: `GET /api/audit/integrity?fresh=1` → `valid: true`, `totalEntries: 11397`, `verifiedEntries: 11397`, `missingHashCount: 0`, `violations: []`.
**UAT Impact**: None — the audit chain is now 100% verified. Testers will see `11397/11397 verified, 0 violations` (the count will grow as UAT creates new audit entries; that is expected and remains `valid: true`).

### Issue 7: Other routes bypass `createAuditLog()` — **TRACKED for v7.3.1 (NOT a UAT blocker)**
**Description**: A codebase audit found that 14+ other API routes create AuditLog entries via direct `db.auditLog.create()` instead of the canonical `createAuditLog()` helper. Some of these compute a `sha256Hash` inline (with varying formulas that may not match the verifier), others omit the hash entirely (producing NULL-hash entries like Issue 6 did). The routes include: `admin/ai-config`, `chat/scenarios`, `remediations`, `kyc-upload`, `kyc/individual`, `kyc/corporate`, `ai/enhanced`, `ai/policy-rag`, `cbuae-submission-checker`, `alerts/status`, `compliance-calendar`, `compliance-alerts`, `sar-deadlines`, `complaints/[id]/communications`.
**Why this is NOT a UAT blocker**: The `/api/audit/integrity` verifier reports `missingHashCount` SEPARATELY from `violations`. The `valid` field is `violations.length === 0` — so NULL-hash entries do NOT make the audit trail "invalid"; they only depress `verifiedEntries` below `totalEntries`. None of these 14 routes are exercised by the 5 UAT test scenarios (they cover complaints, GoAML, policies, attestations, data-room, my-tasks — none of which touch the 14 listed routes except attestations, now fixed). The current state is `11397/11397 verified, 0 missing, 0 violations`.
**Mitigation during UAT**: If a tester exercises one of these routes and a new NULL-hash entry appears, run the idempotent backfill to restore full verification:
```bash
node scripts/backfill-audit-hashes.mjs   # recomputes hashes for any NULL/mismatched entries
curl -s -H "x-user-id: u" -H "x-user-role: admin" "http://localhost:3000/api/audit/integrity?fresh=1"
```
**v7.3.1 remediation**: Migrate all 14 routes to use `createAuditLog()` (single canonical path for hash computation + PII sanitization). This eliminates the divergence risk permanently.

## Integration Simulation Mode

> Added by UAT Hotfix Batch 4 (Issue 11). All 3 external integration adapters
> are **fully implemented** with real API call paths, but default to
> **SIMULATION mode** during UAT/dev because no production credentials are
> configured. This is **expected and acceptable for UAT** — simulation
> responses are clearly tagged so testers do not report them as bugs.

### What "simulation mode" means

Each adapter emits a one-time `SIMULATION MODE ACTIVE` console warning on first
use, then returns tagged mock data:
- **goAML regulatory-gateway** → `mode:'simulation'`, `fiuReceiptNumber:'FIU-SIM-...'`
- **screening-provider** (Dow Jones / Refinitiv) → `mode:'simulation'`, synthetic PEP/sanctions hits
- **identity-provider** (UAE Pass / Nafath) → `mode:'simulation'`, `accessToken:'sim-access-token-...'`

The simulation paths exercise the full workflow (state transitions, audit
logging, receipt capture) — only the external network call is mocked.

### How to switch an adapter to PRODUCTION (live) mode

Set the environment variables for the adapter(s) you want to go live. Each
adapter's source file has a `CONFIGURATION (Simulation vs Production)` block
documenting its exact vars (see `src/lib/integrations/*.ts`).

| Adapter | Env vars to set for LIVE mode |
|---------|-------------------------------|
| **goAML FIU submission** (`regulatory-gateway.ts`) | `GOAML_SUBMISSION_MODE=direct_api` + `GOAML_API_KEY` + `GOAML_CERT_PATH` + `GOAML_CERT_PASSWORD` (mTLS). `GOAML_API_URL` optional (default `https://fiu.goaml.gov.ae/api/v2`). |
| **Screening** (`screening-provider.ts`) | `SCREENING_API_URL` + `SCREENING_API_KEY`. `SCREENING_PROVIDER=refinitiv\|dowjones` optional. |
| **Identity / SSO** (`identity-provider.ts`) | `IDENTITY_API_URL` + `IDENTITY_API_KEY`. `IDENTITY_PROVIDER=uae_pass\|nafath` optional (default `uae_pass`). |

### UAT guidance

- **Do NOT report simulation-mode responses as bugs.** They are the documented
  default for UAT. The `mode:'simulation'` tag in the response confirms the
  adapter is working correctly in its non-production configuration.
- If a UAT scenario requires **live** integration testing (e.g. a real FIU
  submission), coordinate with the platform admin to provision credentials and
  set the env vars above. Otherwise, simulation mode is sufficient for all 5
  UAT test scenarios.
